# /onboard — 워크스페이스 건축

이 스킬은 **워크스페이스를 처음 세우는** 1회성 파이프라인이다. 운전은 `workspace-architect` 에이전트가 한다.

## 절차

1. **건축 에이전트 호출** — Agent 도구로 `workspace-architect` 를 실행. 인터뷰를 위임한다.
   (전체 정의: `.claude/agents/workspace-architect.md`)
2. 건축가가 진행하는 것 (DESIGN §6-A):
   - **인터뷰** 6~8문항 (한 번에 하나씩, 자명하면 건너뛰기) — 직무·도메인·산출물형태·조력자수·톤·이름·명명규약
   - **생성** — `seed.json` 갱신(이름·PREFIX·slug_lang) → `python bin/ws.py init` → `CLAUDE.md`·`PROGRAM.md` 의 `{{...}}` 치환 → 조력 에이전트 1~2명을 `.claude/agents/` 에 작성
   - **시연** — 첫 업무 1개를 `ws.py add` → `ws.py deliver` 로 md+html 산출물 생성. 산출물 형태는 인터뷰에서 답한 주 산출물에 맞춘다(아래 "첫 산출물 형태" 참고).
   - **점화** — `deliverables/_dashboard.html` 을 열어 보여주고 "여기서 시작하세요" 안내 후 종료
3. **멱등성** — 이미 구축된 워크스페이스에서 재실행되면 덮어쓰지 말고 누락분만 보완.

## 첫 산출물 형태 (검증된 골격)

첫 시연 산출물은 사용자의 주 산출물에 맞춰 빚되, 아래 형태 중 하나를 기본으로 제안한다:

- **보고서 / 리서치 노트** — 제목 → 요약 → 본문 섹션 → 결론
- **회의록** — 결정 사항 → 액션 아이템 → 다음 안건
- **체크리스트 / 가이드** — *짧은 문항 몇 개 → 한 장 정리*. 실사용 검증된 형태(예: "○○ 전 4가지만 확인" 식 사전 점검 가이드). 읽는 사람이 순서대로 따라가며 채우는 구조라, 남에게 건네는 산출물에 특히 강하다.

핵심은 *"저장하면 md+html + 대시보드로 즉시 뜬다"* 를 첫 산출물에서 체감시키는 것. 형태보다 **완주(브라우저에 뜸)** 가 우선.

## 대시보드 골격 — 설정하지 않는다, 가리키기만 한다

온보딩은 대시보드 **골격(레이아웃)을 설정하지 않는다.** 기본은 항상 테이블(`_dashboard.html`)이고, `render.py`·`seed.json` 스키마는 건드리지 않는다. (원탁 4회차 결정 — render 레이아웃 추가 = 표면 확장, 굳히기 국면 각하.)

대신 점화 단계에서, 인터뷰 4번(주 산출물 형태) 답에 맞춰 **참고 포인터 한 줄**만 남긴다(코드 변경 없음):

| 주 산출물 형태 | 어울리는 골격 (참고) |
|---|---|
| 보고서·지표 | `docs/dashboard-catalog/06-insight-deck.html` (차트) |
| 회의록·과제 흐름 | `04-kanban-flow.html` (칸반) · `02-workspace-views.html` |
| 결과물 보여주기 | `05-card-gallery.html` (갤러리) |
| 일정 중심 | `07-roadmap-timeline.html` (타임라인) |

안내 예: *"기본 대시보드는 테이블이에요. 진행 흐름 중심이시면 `docs/dashboard-catalog/04-kanban-flow.html` 골격이 어울립니다 — 이 10종은 쇼케이스라 자동 적용은 안 되고, 원할 때 HTML을 참고해 수동 채택하시면 됩니다."*

## 끝나고

사용자에게: 이후 일상은 `/task`(업무) · `/deliver`(산출물) · `/view`(대시보드) 로 돌린다고 알린다.
